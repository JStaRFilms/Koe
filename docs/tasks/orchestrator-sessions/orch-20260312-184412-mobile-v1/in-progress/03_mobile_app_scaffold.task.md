## Objective

Create the initial Expo Router application shell for Koe mobile so later tasks can add recording, settings, and history without restructuring the app.

## Scope

**In scope:**
- Expo Router app shell
- Base navigation/layout
- Recorder home route placeholder
- Settings route placeholder
- History route placeholder
- Shared visual tokens and app chrome
- Development build readiness for iOS and Android

**Out of scope:**
- Final recording implementation
- Groq integration
- History persistence logic
- Android IME

## Product Shape

The agreed V1 mobile UX is:
- app-first, not keyboard-first
- one primary recorder screen
- settings and history inside the app
- clipboard-first completion flow

## Required Screens And Structure

Use a route structure that supports:

```text
apps/mobile/app/
  _layout.tsx
  index.tsx          // recorder home
  history.tsx
  settings.tsx
  onboarding.tsx     // if first-run flow is needed
```

If route groups improve structure, use them, but keep the resulting navigation simple.

## UI Requirements

- Clear recorder-first home screen
- Native-feeling navigation and safe-area handling
- A visible state model for idle, recording, processing, copied, and error
- Styling should feel consistent with Koe but adapted to mobile rather than copying the desktop pill
- Prefer bold, deliberate mobile UI over a generic scaffold

## Technical Requirements

- Use Expo Router and TypeScript
- Set up app metadata, package name placeholders, and sensible defaults for iOS/Android
- Add the packages later tasks need if they are low-risk foundational dependencies:
  - `expo-router`
  - `expo-clipboard`
  - `expo-secure-store`
  - `expo-haptics`
  - `expo-linking`
- Add any setup needed for React Native Reanimated if the chosen UI uses it
- Keep the shell compatible with development builds, not just Expo Go
