## Objective

Implement the first functional mobile recording pipeline: capture audio, segment it safely, send it to Groq directly with the user key, refine text, track usage, and support retry paths.

## Scope

**In scope:**
- microphone permission flow
- recording start/stop
- chunk or session buffering
- direct Groq calls from mobile
- refined transcript generation
- error handling and retry
- shared-core integration
- background-safe behavior where supported

**Out of scope:**
- Android keyboard service
- managed Koe accounts
- server-side proxying

## Product Decisions To Respect

- V1 uses **manual start/stop recording**
- V1 is **clipboard-first**
- The user provides a Groq key locally
- The pipeline should favor reliability over trying to clone desktop VAD behavior on day one

## Technical Requirements

### Recording Model

- Implement a recorder controller in mobile that supports:
  - start
  - stop
  - flush/finalize
  - cancellation
- Prefer a manual session model with safe chunking for longer recordings
- If chunking is required during long recordings, keep ordering deterministic
- Do not make VAD a hard dependency for V1

### Provider Integration

- Use the shared provider/session contracts from `packages/koe-core`
- Support:
  - transcription
  - text refinement
  - usage accounting
  - failure state propagation
- Store the Groq key securely on device

### Reliability

- Handle invalid key, network failure, empty transcript, and partially failed processing
- Preserve enough local state to retry a failed session or segment
- Do not silently discard captured speech

### App Lifecycle

- Request and explain microphone permissions clearly
- Configure background recording/processing support where Expo/native support allows it
- If the OS suspends execution, surface a clear failure or resume state instead of pretending work finished
