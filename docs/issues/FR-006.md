# [FR-006] Rate Limiting

## Labels
`MUS`, `enhancement`, `groq-integration`

## User Story
As a user, I want the app to respect Groq's 20 RPM / 2K RPD limits, so my API key isn't banned.

## Proposed Solution

### Overview
Implement a sliding window rate limiter that enforces Groq's free tier limits. Queue requests that exceed limits and process them when slots open.

### Implementation Flow
1. Track requests with timestamps in a sliding window (last 60 seconds for RPM)
2. Track daily request count and audio seconds in electron-store (reset at midnight)
3. Before each API call: check RPM, RPD, and audio seconds
4. If under limits: proceed immediately
5. If over limits: queue the chunk, process when a slot opens
6. Expose usage stats to UI via IPC

### Key Considerations
- Three limits to track: 20 RPM, 2,000 RPD, 28,800 audio sec/day
- Midnight reset should use local time
- Queue should be FIFO — chunks processed in order
- If daily limit is hit: inform user, disable recording until next day
- Persist daily stats across app restarts via electron-store

## Acceptance Criteria

- [ ] Requests are limited to 20 per minute
- [ ] Requests are limited to 2,000 per day
- [ ] Audio seconds are tracked against 28,800/day limit
- [ ] Excess requests are queued and processed when slots open
- [ ] Daily stats persist across app restarts
- [ ] User is notified when approaching/hitting limits
